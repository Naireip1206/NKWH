# 服务端的相关经验

> 个人从 Minecraft 租赁服到自主搭建服务器全过程的经验记录。
> 原始内容由用户撰写，经 LLM 扩充整理。

---

## 一、起点：网易租赁服

我一开始是在网易我的世界里买的**租赁服**玩的。

那是大多数人入坑联机的最便捷途径——不用管任何技术问题，付款即玩。但限制也非常明显：

- ❌ **不能导出存档** — 一旦停服，数据就全没了，无法迁移到其他平台
- ❌ **不能加任何模组和插件** — 只能玩原版，没有任何扩展空间
- ❌ **月租不便宜** — 以网易当时的定价，拿到的配置和功能都非常有限，性价比很低

这个阶段让我意识到：想真正掌控自己的 Minecraft 世界，必须自己搭服务器。

---

## 二、第一步尝试：基岩版官方服务端

后来我开始研究开一个**原版服务器**，使用的是 **基岩版官方服务端（Bedrock Dedicated Server，简称 BDS）**。

当时的硬件条件是：**1 核 2G 的云服务器**。

- 运行结果：**非常卡**，稍微有几个玩家就撑不住了
- 原因分析：BDS 本身优化一般，加上 1c2g 的小水管同时跑系统和 MC 服务端，资源严重不足
- 但这仍然是我迈向**自主构建服务器**的第一步——第一次自己下载服务端、配置参数、开放端口让朋友进来玩

这个阶段虽然体验不好，但学到了最基础的服务端运维概念：端口转发、配置文件、日志排查等。

---

## 三、Java 版服务端的探索之路

在基岩版之后，我转向了 **Java Edition**，开始了漫长的服务端核心探索之旅。

### Paper → Mohist → Purpur

- **Paper** — 当时 Java 版最主流的优化服务端，基于 Spigot/Bukkit API，性能和稳定性都远超原版 Vanilla。是我用的第一个 Java 版服务端核心。
- **Mohist** — 尝试它是因为想同时跑 Bukkit 插件和 Forge Mod。Mohist 是一个混合核心（Hybrid Core），试图桥接插件和 Mod 两个生态。但实际体验中稳定性不如纯血核心，后来就没有继续使用。
- **Purpur（朋友推荐）** — 朋友推荐我试试 Purpur。它基于 Paper，额外提供了大量可配置的**游戏机制调整选项**，比如修改玩家行为、调整实体 AI、自定义堆叠上限等。在不装插件的情况下就能做很多定制。这让我认识到：**服务端核心的选择不仅仅是性能问题，更是功能和灵活度的问题**。

### Fabric + 优化模组（至今推荐）

后来我尝试了 **Fabric + 优化模组** 的方案，获得了非常好的性能收益。

核心思路是：用 Fabric（一个轻量级的 Mod 加载器）搭配专门的**优化模组**，而不是用 Paper 那种大改服务端内部实现的方案。这样既能获得接近 Paper 的性能优化，又能保持 Fabric 生态的灵活性和兼容性。

我使用的主要优化模组包括：

| 模组 | 作用 |
|------|------|
| **Lithium** | 优化游戏引擎的通用计算逻辑，减少不必要的计算开销 |
| **Phosphorus**（原 Phosphor） | 优化光照引擎，大幅降低光照更新带来的卡顿 |
| **Starlight** | 另一款光照优化模组，重写了光照引擎，性能提升非常显著 |
| **Krypton** | 优化网络栈，减少网络延迟和带宽占用 |
| **FerriteCore** | 优化内存布局，减少内存占用 |
| **ModernFix** | 综合性能优化，修复了很多原版的性能问题 |
| **No Chat Reports** | 不完全是优化，但对私服很重要，禁止聊天上报 |

**这套方案我至今也非常推荐**，特别是对于想要保持原版体验或者想同时加少量 Mod 的服务器。它的优势在于：

1. 优化效果显著，TPS 稳定在 20
2. 兼容性好，不像大改核心那样容易出兼容性问题
3. 按需选择优化模组，不需要的可以不装
4. Fabric 生态轻量，启动速度快

### Folia 及其 Fork

再后来我开始接触 **Folia**，这是一个划时代的服务端核心。

Folia 由 Paper 团队开发，但它和 Paper 有着本质的不同：

- Paper 仍然是**单线程主循环**架构（虽然部分任务异步化）
- Folia 将世界划分为多个 **Region（区域）**，每个 Region 可以独立地在**不同的线程**上运行 tick 循环
- 这意味着：**多核 CPU 的优势终于能被 MC 服务端真正利用起来**

Folia 的架构突破：

```
你想开什么类型的服？
├─ 原版/插件生存服
│   ├─ 传统单线程 → Paper 系
│   │   ├─ 需要定制游戏机制 → Purpur
│   │   └─ 稳定可靠 → Paper 本身
│   └─ 多线程架构 → Folia 系
│       ├─ 追求极致多线程性能 → Folia / Leaf
│       ├─ 激进性能优化 → Lumino
│       ├─ 兼顾兼容性 → Leaves
│       └─ 精简轻量 → Mint
├─ 轻量优化 + Mod → Fabric + 优化模组 ★ 推荐
└─ 大型 Mod 整合包 → Forge / NeoForge（整合包自带）
```

Folia 的分区机制（Region-Based Multi-Threading）带来了显著的性能提升：

- **高密度实体场景**：不同区域的实体在不同的线程上驱动，不会互相阻塞
- **多世界架构**：不同世界可以并行 tick，不再串行等待
- **大型服务器**：几十甚至上百玩家的场景下，Folia 的表现远超传统核心

但 Folia 也有代价：
- 很多插件不兼容 Folia（因为很多插件假设只有一个主线程）
- 需要插件和开发者针对 Folia 的分区架构做适配
- 小规模服务器（个位数玩家、单世界）感受不到明显优势

#### Folia 的 Fork

随着 Folia 的出现，社区中也出现了一些基于 Folia 的 Fork：

- **Leaf** — 轻量级 Folia Fork，专注于兼容性和易用性
- 其他 Fork 也在不断涌现，进一步发扬了 Folia 的多线程思路

> 以上都是原版/插件服的构建经验。Folia 代表了 MC 服务端性能优化的下一个方向，虽然目前生态还不够成熟，但它的架构理念将会影响未来所有服务端核心的发展。

## 四、Mod 服的经历

在这之后我开始开 **Mod 服**（模组服），但绝大多数情况下用的是**整合包自带的服务端**。

知名的整合包（如 ATM、Enigmatica、GT:NH 等）通常都会提供预配置的服务端文件：
- 服务端核心（通常是 Forge 或 NeoForge）
- 完整的 Mod 列表和配置
- 启动脚本
- 有时候还附带优化建议

服主只需要：
1. 上传文件到服务器
2. 配置端口和内存
3. 启动即可

**并不需要由服主本人操作太多，这里就不再赘述了。**

不过有一点值得注意：Mod 服的性能瓶颈通常不在服务端核心的选择上，而在于 Mod 本身的质量。一个写得不好的 Mod 可以把任何服务端核心都拖垮。因此 Mod 服优化的关键在于：
- 选择优化良好的整合包
- 合理分配内存（Mod 服通常需要更多内存，建议 6G-12G）
- 使用 Spark 等工具定位是哪个 Mod 导致的卡顿

## 五、推荐的端（服务端核心）

综合以上所有经验，我推荐以下几个服务端核心：

### Purpur
- **基于 Paper**，兼容 Bukkit/Spigot/Paper 插件生态
- **特点**：大量可定制的游戏机制选项，适合想要精细控制游戏体验的服主
- **适用场景**：主流插件生存服、小游戏服、不需要 Mod 的几乎所有场景
- **推荐理由**：稳定、功能丰富、社区活跃

### Lumino（原 Airplane + Pufferfish 的延续）
- **基于 Folia**，融合了 Pufferfish 和 Airplane 的优化思路
- **特点**：引入了多项实验性优化，包括实体激活优化、动态 tick 速率等
- **适用场景**：高玩家负载的生存服、追求极致性能的场景
- **注意**：部分优化可能导致红石机械行为异常，需要测试

### Leaves
- **基于 Folia**，同时兼容部分 Bukkit/Paper 插件生态
- **特点**：在 Folia 多线程架构的基础上，努力改善插件兼容性
- **适用场景**：想尝试 Folia 多线程优势但又担心插件兼容性的场景

### Mint
- **基于 Folia**，专注于轻量和简洁
- **特点**：在 Folia 的基础上去除许多非必要特性，保持最小化核心，让服主自己按需添加
- **适用场景**：喜欢折腾、追求精简、想要最小攻击面的服主

## 六、经验总结

以下是这段时间折腾下来的一些核心感悟：

### 选择服务端核心的决策路径

```
你想开什么类型的服？
├─ 原版/插件生存服
│   ├─ 传统单线程 → Paper 系
│   │   ├─ 需要定制游戏机制 → Purpur
│   │   └─ 稳定可靠 → Paper 本身
│   └─ 多线程架构 → Folia 系
│       ├─ 追求极致多线程性能 → Folia / Leaf
│       ├─ 激进性能优化 → Lumino
│       ├─ 兼顾兼容性 → Leaves
│       └─ 精简轻量 → Mint
├─ 轻量优化 + Mod → Fabric + 优化模组 ★ 推荐
└─ 大型 Mod 整合包 → Forge / NeoForge（整合包自带）
```

### 硬件建议
- **CPU**：单核性能 > 核心数。高频 CPU（如 Intel 的 K 系列、AMD X3D 系列）优势明显
- **内存**：原版/插件服 4G-8G 起步，Mod 服 8G-16G
- **硬盘**：NVMe SSD 是必须的，区块加载速度直接影响玩家体验
- **云服务器**：轻量服用 4c8g 起步，Mod 服建议 8c16g 或更高
  - 国内推荐：腾讯云轻量、阿里云轻量（性价比高）
  - 国外推荐：OVH、Hetzner（性价比极高）、AWS Lightsail

### 优化方法论
1. **先定位瓶颈，再优化** — 用 Spark、/mspt 等工具找出真正的瓶颈
2. **不要盲目加优化** — 不必要的优化可能引入兼容性问题
3. **预生成区块** — 用 Chunky 在开服前生成世界，是最简单有效的优化手段
4. **控制视距** — view-distance=6 是最平衡的值
5. **限制实体** — 掉落物合并、村民数量控制、动物上限设置

### 踩坑记录
- ❌ **不要用 1c2g 跑任何 MC 服务端** — 连开服都勉强
- ❌ **不要在小服上盲目上 Folia** — 单线程就够的情况下，Folia 只会增加复杂性
- ❌ **不要混合核心** — Mohist 之类的混合核心在稳定性和兼容性上总有妥协
- ✅ **Fabric + 优化模组是中小型服务器的最优解**
- ✅ **用 Systemd 管理服务端进程** — 崩溃自动重启，省心很多
- ✅ **定期备份是最重要的运维习惯** — crontab + 压缩 + 异地存储

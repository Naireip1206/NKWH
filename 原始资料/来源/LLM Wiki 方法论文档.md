## 核心思想

大多数人与 LLM 和文档的交互方式是 RAG：上传文件集合，LLM 在查询时检索相关片段并生成答案。这种方式可行，但 LLM 在每个问题上都在重新发现知识——没有积累。问一个需要综合五份文档的微妙问题，LLM 每次都得寻找并拼凑相关片段。什么也没有建立起来。NotebookLM、ChatGPT 文件上传和大多数 RAG 系统都是这样运作的。

这里的思路不同。LLM 不是仅在查询时从原始文档检索，而是**增量式构建并维护一个持久的维基**——一个结构化的、相互链接的 Markdown 文件集合，位于你和原始资料之间。当你添加新来源时，LLM 不只是为后续检索做索引——它读取内容、提取关键信息、并将其整合到现有维基中——更新实体页面、修订主题摘要、标注新数据与旧主张的矛盾之处、加强或质疑正在演化的综合。知识被编译一次之后就**保持当前**，而不是在每个查询上重新推导。

这是关键区别：**维基是一个持久的、不断叠加的产物。** 交叉引用已经存在，矛盾已经被标注，综合已经反映了你读过的一切。每个你添加的来源和提出的问题都让维基变得更加丰富。

你从不（或很少）自己写维基——LLM 编写和维护所有内容。你负责策展、探索和提出正确的问题。LLM 做所有繁琐的工作——总结、交叉引用、归档和簿记——这些工作让知识库随着时间推移真正有用。实践中，我让 LLM 代理在一边打开，Obsidian 在另一边打开。LLM 根据我们的对话进行编辑，我实时浏览结果——跟随链接、检查图谱视图、阅读更新后的页面。Obsidian 是 IDE，LLM 是程序员，维基是代码库。

这可以应用于许多不同场景：
- **个人**：追踪自己的目标、健康、心理、自我提升——归档日记、文章、播客笔记，随时间构建关于自己的结构化图景
- **研究**：在数周或数月内深入研究一个主题——阅读论文、文章、报告，增量式构建一个包含演化论点的全面维基
- **读书**：逐章归档，为角色、主题、情节线索及其关联建页面。读完后你得到一个丰富的伴读维基
- **商业/团队**：由 LLM 维护的内部维基，以 Slack 讨论、会议转录、项目文档、客户通话为输入
- **竞品分析、尽职调查、旅行规划、课程笔记、爱好深潜**——任何随时间积累知识并希望其有条理而非散落的场景

## 架构

**原始资料** — 你策展的原始文档集合。文章、论文、图片、数据文件。不可变——LLM 从中读取但从不修改。这是事实源。

**维基** — 一个 LLM 生成的 Markdown 文件目录。摘要、实体页面、概念页面、比较、概览、综合。LLM 完全拥有这一层。它创建页面、在新来源到达时更新、维护交叉引用、保持一切一致。你阅读它；LLM 编写它。

**模式文档** — 一个文档（如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md）告诉 LLM 维基的结构、约定和工作流程。这是关键的配置文件——它使 LLM 成为一个有纪律的维基维护者而非通用聊天机器人。你和 LLM 随时间共同演化它。

## 操作

**摄取（Ingest）**：放入新来源到原始集合，告诉 LLM 处理。示例流程：LLM 读取来源→与你讨论关键收获→在维基中写摘要页面→更新索引→更新相关实体和概念页面→追加日志条目。一个来源可能触及 10-15 个维基页面。建议一次处理一个，保持参与——你阅读摘要、检查更新、指导 LLM 重点方向。也可以批量处理。

**查询（Query）**：向维基提问。LLM 搜索相关页面、阅读、综合带引用的回答。回答可以是 Markdown 页面、对比表格、幻灯片、图表等。**好的回答可以归档回维基成为新页面。** 这样你的探索像摄入的来源一样叠加在知识库中。

**维护（Lint）**：定期让 LLM 做健康检查——查找页面间的矛盾、被新来源取代的过时主张、无入站链接的孤立页面、被提及但缺独立页面的重要概念、缺失的交叉引用、可通过网络搜索填充的数据缺口。

## 索引与日志

**索引.md** 是内容导向的目录。每页一条+链接+一句话摘要+元数据。按类别组织。每次摄取后更新。在中等规模（~100来源，~数百页面）下效果很好，避免了对嵌入检索基础设施的需求。

**日志.md** 是时间顺序的记录。仅追加的操作记录——摄取、查询、维护。如果每条以统一前缀开头，日志就可用简单 Unix 工具解析。

## 可选工具

- **Obsidian Web Clipper** — 浏览器扩展，将网页转为 Markdown
- **qmd** — Markdown 文件的本地搜索引擎，支持混合 BM25/向量搜索和 LLM 重排序
- **Obsidian 图谱视图** — 查看维基形状的最佳方式
- **Marp** — 基于 Markdown 的幻灯片格式
- **Dataview** — 对页面 frontmatter 运行查询的 Obsidian 插件
- **Git** — 版本历史、分支和协作

## 为什么有效

维护知识库最繁琐的不是阅读或思考——是簿记。更新交叉引用、保持摘要最新、标注新数据与旧主张的矛盾、在数十个页面间保持一致性。人类放弃维基是因为维护负担增长快于价值。LLM 不会感到无聊、不会忘记更新引用、可以一次处理 15 个文件。维基保持维护因为维护成本接近于零。

人类的工作是策展来源、指导分析、提出好问题、思考意义。LLM 的工作是其他一切。

这个思路与 Vannevar Bush 的 Memex（1945）在精神上一脉相承——一个私人的、积极策展的知识存储，文档之间有联想性路径。Bush 的愿景更接近这个模式而非后来的网络：私人的、主动策展的、文档之间的联系本身与文档同样有价值。他没能解决的部分是谁来做维护。LLM 接手了。

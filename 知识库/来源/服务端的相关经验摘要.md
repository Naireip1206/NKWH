# 服务端的相关经验摘要

> 来源：[[原始资料/资产/服务端的相关经验.md]] — 用户撰写的 Minecraft 服务端搭建个人经验记录。

## 概述

本文记录了用户从网易租赁服到自主搭建 Minecraft 服务端的完整经验历程。涵盖了租赁服的局限性、基岩版官方服务端的初次尝试、Java 版多种服务端核心（Paper/Mohist/Purpur/Fabric/Folia）的探索过程，以及最终的服务端核心推荐。核心主张是：**Fabric + 优化模组是中小型服务器当前的最优解**。

## 关键要点

### 经历阶段

| 阶段 | 使用方案 | 评价 |
|------|---------|------|
| 起步 | 网易租赁服 | 限制多、价格高、不能导出存档 |
| 启蒙 | 基岩版官方服务端（BDS）+ 1c2g | 很卡，但学会了基础运维 |
| 探索 | Paper → Mohist → Purpur | 在性能和功能之间逐步找到平衡 |
| 成熟 | Fabric + 优化模组（Lithium/Phosphorus/Krypton等） | **至今强烈推荐**，性能好、兼容性高 |
| 前沿 | Folia 及其 Fork（Leaf 等） | 多线程架构是未来方向，但生态尚不成熟 |
| Mod 服 | 整合包自带服务端（Forge/NeoForge） | 开箱即用，但性能瓶颈常在 Mod 本身 |

### 推荐的服务端核心

- **Purpur** — 基于 Paper，大量可定制机制，适合主力插件生存服
- **Folia / Leaf** — 多线程架构，适合大型服务器
- **Leaves** — 基于 Folia，兼顾兼容性
- **Lumino** — 基于 Folia，融合 Pufferfish/Airplane 的激进优化
- **Mint** — 基于 Folia 的轻量精简核心

### Fabric + 优化模组方案推荐配置

| 模组 | 作用 | 推荐程度 |

### Folia 的架构创新

Folia 将世界划分为多个 Region（区域），每个 Region 可在不同线程上独立 Tick，首次让 MC 服务端真正利用多核 CPU。但代价是插件兼容性问题。

### 硬件建议

- **CPU**：单核性能 > 核心数
- **内存**：原版/插件服 4G-8G，Mod 服 8G-16G
- **硬盘**：NVMe SSD 必须
- **云服务器**：4c8g 起步（轻量服），8c16g 起步（Mod 服）

### 核心避坑建议

1. ❌ **不要用 1c2g 跑任何 MC 服务端**
2. ❌ **小服不要盲目上 Folia**（增加复杂性但无明显收益）
3. ❌ **混合核心（如 Mohist）稳定性有妥协**
4. ✅ **Fabric + 优化模组是中小型服务器的最优解**
5. ✅ **Systemd 管理进程 + 定期备份是基本运维要求**
6. ✅ **预生成区块是最简单有效的优化手段**

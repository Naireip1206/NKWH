# 服务器性能优化

## 定义

MC 服务器性能优化是指通过调整 JVM 参数、服务端配置、硬件选择和软件架构等手段，使 MC 服务器在保持稳定 20 TPS 的前提下，尽可能承载更多玩家和更复杂的游戏机制。

## 优化维度

### 1. JVM 参数优化（Aikar's Flags）

MC 服务器的 Java 进程对垃圾回收（GC）非常敏感——GC 引发的 STW（Stop-The-World）暂停会直接体现为玩家感受到的卡顿。Aikar's Flags 是目前业界公认的最佳 JVM 参数集合：

```
-Xms4G -Xmx8G -XX:+UseG1GC -XX:+ParallelRefProcEnabled
-XX:MaxGCPauseMillis=200 -XX:+DisableExplicitGC
-XX:+AlwaysPreTouch -XX:G1HeapWastePercent=5
-XX:G1MixedGCCountTarget=4 -XX:G1MixedGCLiveThresholdPercent=90
-XX:G1RSetUpdatingPauseTimePercent=5 -XX:SurvivorRatio=32
-XX:MaxTenuringThreshold=1 -XX:+PerfDisableSharedMem
```

**核心思路**：
- G1GC 在延迟和时间确定性上优于默认的 Parallel GC
- `AlwaysPreTouch` 消除运行时的内存申请延迟
- `DisableExplicitGC` 防止插件误触发 Full GC
- `MaxTenuringThreshold=1` 减少晋升管理的开销

### 2. 服务端核心选择

| 核心 | 相对性能（以 Vanilla 为基线） | 说明 |
|------|------------------------------|------|
| Vanilla | 1.0x | 无任何优化 |
| Spigot | ~1.3x | 少量优化 |
| **Paper** | **~1.8x** | **显著优化，推荐首选** |
| Purpur | ~1.8x | 基于 Paper，更多可调配置 |
| Pufferfish | ~1.9x | 针对高实体密度优化 |
| Fabric + Lithium | ~1.6x | 优化 + Mod 兼容性 |

### 3. 世界与区块管理

- **预生成区块**：使用 Chunky 或类似工具在开服前生成足够大的区块区域，避免玩家探索时实时生成导致卡顿
- **降低视距**：`view-distance=6` / `simulation-distance=5` 是多数服务器的有效折中
- **实体数量控制**：过多的掉落物、村民、动物是常见的 TPS 杀手
- **红石机器限流**：高频红石机器（刷沙机、分类机等）是单线程性能的天敌

### 4. 内存管理

- **Xms = Xmx**：避免 GC 调整堆大小
- **不要过度分配**：内存过多反而导致 GC 暂停变长（更大的堆 = 更长的 GC 周期）
- **监控 Old Gen 使用率**：Old Gen 不断增长可能是内存泄漏的信号

### 5. 插件与 Mod 层面的优化

| 目标 | 推荐工具 |
|------|---------|
| 定时清理 | ClearLag（定期清理掉落物和实体） |
| 区块加载控制 | ChunkyBorder（限制世界边界） |
| 性能分析 | Spark（精确定位卡顿原因） |
| 光照优化 | Starlight / Phosphorus（Fabric） |
| 网络优化 | Krypton（Fabric） |

## 性能诊断手段

```bash
# 服务器内命令
/tps            # 查看当前 TPS（> 19 正常）
/mspt           # 查看每 tick 耗时（< 50 优秀）
/paper timings  # Paper 深度性能分析
/spark profiler # Spark 性能采样

# 系统命令
htop            # CPU 和内存实时
iotop           # 磁盘 IO
journalctl -u minecraft -f  # Systemd 日志
```
